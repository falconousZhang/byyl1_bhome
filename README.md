# 类Rust编译器——词法与语法分析器

同济大学编译原理课程大作业1

---

## 项目简介

本项目实现了一个**类Rust语言**的词法分析器与语法分析器，并提供网页前端进行可视化分析。

| 模块 | 方法 | 说明 |
|------|------|------|
| 词法分析 | 手写 DFA | 逐字符扫描，精确记录行列号 |
| 语法分析 | PLY（LALR(1)） | 自动构建 LALR(1) 分析表 |
| 语义分析 | 符号表 + 遍历 AST | 检查三条羁绊约束 |
| 前端 | Flask + 原生 HTML/JS | Token 流、可折叠 AST 树、错误信息 |

支持的语法规则覆盖 **0.1 ~ 9.3** 全部条目（含数组、元组、引用、块表达式、循环表达式等），并实现以下语义约束检查：

- **羁绊A**：赋值时类型不匹配报错
- **羁绊B**：`for` 循环的可迭代对象必须是数组或范围，否则报错
- **羁绊C**：对不可变变量赋值报错，`break` / `continue` 出现在循环外报错

## 语法分析方法说明

### 使用 LALR(1) 而非 LR(1)

本项目使用 **LALR(1)**（通过 PLY 库实现），而非手写的完整 LR(1)。两者关系如下：

```
LR(1)  ⊃  LALR(1)  ⊃  SLR(1)
能力最强        本项目         能力最弱
```

LALR(1) 通过合并核心相同的 LR(1) 状态来压缩状态数，少数情况下会产生 LR(1) 没有的 reduce-reduce 冲突。对本项目的文法，PLY 构建过程**零冲突警告**，说明该文法在 LALR(1) 上与 LR(1) 等价，分析结果完全一致。

### 文法中的三处冲突及解决方案

#### 冲突一：悬空 else（Shift-Reduce Conflict）

**问题**：解析到 `if expr { } else` 时，`else` 应归属当前 if 还是外层 if？

```
if a>0 {
    if a>1 {
        return 1;
    }           ← 看到 else，应该归属哪个 if？
} else {
    return 0;
}
```

**解决**：在优先级表中声明 `ELSE` 优先级高于"无 else 的 if"（虚拟标记 `IFX`），让 LALR(1) 在看到 `else` 时选择 shift（归属最近的 if），符合直觉语义：

```python
precedence = (
    ('nonassoc', 'IFX'),    # 虚拟标记，代表"无 else"，优先级低
    ('nonassoc', 'ELSE'),   # ELSE 优先级更高，触发 shift
    ...
)

def p_if_stmt_no_else(p):
    """if_stmt : IF expr func_body %prec IFX"""  # 显式标记低优先级
```

#### 冲突二：`*` 的双重身份（Shift-Reduce Conflict）

**问题**：`*` 既是中缀乘法运算符（`a * b`），又是前缀解引用运算符（`*a`），分析器需要区分。

**解决**：为前缀用法引入虚拟优先级标记 `UDEREF`，使其优先级高于中缀乘法：

```python
precedence = (
    ...
    ('left',  'STAR', 'SLASH'),          # 中缀乘法，左结合
    ('right', 'UMINUS', 'UDEREF', ...),  # 前缀解引用，右结合，优先级更高
)

def p_expr_deref(p):
    """expr : STAR expr %prec UDEREF"""  # 前缀 * 用 UDEREF 标记
```

同一个 `STAR` token，根据它出现的位置匹配不同的产生式，优先级声明消除二义。

#### 冲突三：两种函数体（需改写产生式）

**问题**：原始文法规定函数体有两种形式：

```
<函数声明> -> <函数头> <语句块>             # 规则 1.1，所有语句以 ; 结尾
<函数声明> -> <函数头> <函数表达式语句块>   # 规则 7.2，末尾可以是裸表达式
```

两者都以 `{` 开头，LALR(1) 合并状态后无法通过优先级区分——**这是 LALR(1)（以及 LR(1)）均无法直接处理的真正冲突**。

**解决：改写产生式，将两种块合并为一个非终结符**，用可选的"尾部裸表达式"统一表达：

```
# 改写前（有冲突）：
func_decl → func_head 语句块
func_decl → func_head 函数表达式语句块

# 改写后（无冲突）：
func_body   →  { body_stmts }

body_stmts  →  ε                    # 空块
body_stmts  →  body_stmts stmt      # 追加一条语句（以 ; 结尾）
body_stmts  →  body_stmts expr      # 末尾追加裸表达式（无 ;）
```

关键在于**第三条产生式**：末尾裸表达式只有在看到 `}` 时才能归约，而带 `;` 的语句归约走第二条。向前看符号 `}` 与 `;` 不同，LALR(1) 凭一个 token 即可唯一决定，冲突彻底消除。

### 三处冲突对比总结

| 冲突 | 类型 | 能否用 LR(1) 直接处理 | 实际解决手段 |
|------|------|------|------|
| 悬空 else | Shift-Reduce | 能（shift 优先） | `%prec IFX` 优先级声明 |
| `*` 双重身份 | Shift-Reduce | 能（优先级区分） | `%prec UDEREF` 优先级声明 |
| 两种函数体 | Reduce-Reduce | **不能**（状态合并后冲突） | **改写产生式**，合并为统一的 `func_body` |

第三条即作业说明中提到的"使得 LR(1) 无法处理的羁绊"——纯靠优先级无法解决，必须修改产生式本身，使文法重新满足 LALR(1) 条件。

## 代码示例

```rust
// 带返回值的函数（规则 1.5）
fn add(a:i32, b:i32) -> i32 {
    return a + b;
}

// if-else if-else（规则 4.3）
fn sign(a:i32) -> i32 {
    if a>0 {
        return 1;
    } else if a<0 {
        return -1;
    } else {
        return 0;
    }
}

// for 范围循环（规则 5.2）
fn sum(mut n:i32) -> i32 {
    let mut s:i32 = 0;
    for mut i in 1..n+1 {
        s = s + i;
    }
    return s;
}

// if 作为表达式（规则 7.3）
fn abs(mut a:i32) -> i32 {
    let mut b = if a>0 { a } else { 0-a };
    return b;
}

// 数组访问（规则 8.3）
fn first(mut a:[i32;3]) -> i32 {
    return a[0];
}
```

## 常见问题

**Q: 提示 `ModuleNotFoundError: No module named 'ply'`**  
A: 确认已激活正确的 conda 环境：`conda activate byyl1`，然后重新运行。

**Q: 浏览器打开后页面空白或报错**  
A: 确认终端中 Flask 已成功启动（显示 `Running on http://127.0.0.1:5000`），检查端口 5000 是否被占用。

**Q: 语法正确但出现语义错误**  
A: 语义错误属于正常检测结果，例如对不可变变量赋值、`for` 循环迭代非数组类型等，参见「语义约束」说明。

**Q: PLY 在 `src/` 目录下生成了 `parser.out` 和 `parsetab.py` 文件**  
A: 这是 PLY 自动生成的分析表缓存文件，属于正常现象，不影响运行。`parser.out` 可用于查看完整的 LALR(1) 状态集，供分析冲突时参考。
